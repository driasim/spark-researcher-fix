# Vibe Vibe Clean Token Product Plan

This document defines the clean website, cohort, and agent posture for the incubator.

Goal:

- keep Spark branded as an incubator and proof engine
- keep token trading on external rails
- give founders and contributors real value before token launch

## Product Posture

Spark should feel like:

- cohort program
- founder operating system
- proof feed
- contributor network
- governance layer

Spark should not feel like:

- a pump board
- a speculative launchpad homepage
- an exchange front-end

## Site Structure

### Public Surfaces

- home page
- batch pages
- project proof pages
- public ship logs
- contributor leaderboard
- reusable asset pages
- governance and treasury reports

### Member Surfaces

- founder workspace
- quest board
- contributor dashboard
- review packets
- token readiness dashboard
- treasury governance surface

### External Trading Surface

Trading should occur off-site on Uniswap-based rails.

Recommended Spark behavior:

- show project proof and readiness
- show route-to-market information
- link outward to supported Uniswap UI / pool pages
- do not embed Spark-native swap execution in v1

## Cohort Flow

1. open applications
2. proof sprint
3. scored admissions packet
4. batch entry
5. weekly build and validation cycle
6. Genesis contribution layer opens
7. token readiness review
8. external launch rail goes live
9. governance and support loops continue

## Project Token Readiness Gate

Every project should pass a readiness gate before public token launch.

Minimum gates:

- product utility exists
- governance scope exists
- founder and treasury packet complete
- trust review passes
- customer or usage proof exists
- clear fee routing and sink design exists
- contribution and claim policy is locked

## Participant Value Surface

Each founder should receive:

- a founder twin
- a startup operating system
- reusable product and growth assets
- proof and content distribution support
- contributor routing
- structured office-hours packets
- token readiness support

Each contributor should receive:

- clear quests
- measurable contribution scoring
- reputation growth
- Genesis Credits
- access to high-signal projects before public token launch

## Viral Surface

The viral loop should come from visible proof:

- what shipped
- what improved
- where contributors can help
- which projects are earning support

The main public object should be the proof page, not the chart.

## Clean Launch Narrative

Recommended story:

- build in public
- contribute early
- earn standing through real value
- unlock token exposure only after the project proves utility

Avoid:

- "buy before everyone else"
- "hold to farm volume"
- "number-go-up first"
