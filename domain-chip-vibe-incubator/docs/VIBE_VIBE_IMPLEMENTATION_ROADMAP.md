# Vibe Vibe Implementation Roadmap

This document defines the pre-build implementation sequence for `Vibe Vibe`.

## Phase 0. Brand And Design Alignment

Deliver:

- `Vibe Vibe` naming alignment in docs and product copy
- design reference based on `Autoresearch Collective`
- direct repo extraction and shell mirror spec from `Autoresearch Collective`
- page inventory
- interaction rules for proof-first surfaces

Success condition:

- the product has one coherent visual and naming direction before engineering expands

## Phase 1. Core Application Shell

Build:

- activity rail
- main workspace surface
- context sidebar
- theme system
- navigation state
- auth and session model

Success condition:

- founders and operators can move through the system with a consistent shell

## Phase 2. Cohort And Founder OS

Build:

- applications
- proof sprint intake
- batch pages
- founder workspace
- office-hours packets
- venture dashboard

Success condition:

- one batch can run through intake, weekly review, and founder ops inside one system

## Phase 3. Contribution And Genesis Layer

Build:

- contributor profiles
- quest board
- Genesis Credits ledger
- evidence review tools
- anti-farm controls
- claim-right generation

Success condition:

- early community participation creates real value and measurable standing

## Phase 4. Token Readiness And Governance

Build:

- token readiness dashboard
- `SPARK` and `veSPARK` views
- bounded futarchy proposal surfaces
- treasury policy screens
- support reserve flows

Success condition:

- governance and support loops exist before public project token launches

## Phase 5. External Launch Routing

Build:

- project launch eligibility checks
- external Uniswap route integration
- proof-to-launch handoff
- post-launch treasury and support monitoring

Success condition:

- `Vibe Vibe` can route a clean graduated project to external market rails without becoming the swap venue

## Phase 6. Portfolio Learning And Public Growth

Build:

- portfolio learning views
- reusable asset exchange
- public proof feed
- contributor and founder status pages
- ecosystem governance history

Success condition:

- the public sees progress, proof, and community value creation clearly

## Phase 7. Hardening

Build:

- trust and abuse controls
- sybil detection
- permissions
- treasury guardrails
- review workflows
- analytics and telemetry

Success condition:

- the system is resilient enough for real user volume and real token governance

## Immediate Pre-Build Deliverables

Before implementation starts, finish:

- design reference
- Collective mirror spec
- token stack docs
- futarchy doc
- Genesis Credits doc
- product plan
- data and contract map

That is the minimum clean architecture set.
