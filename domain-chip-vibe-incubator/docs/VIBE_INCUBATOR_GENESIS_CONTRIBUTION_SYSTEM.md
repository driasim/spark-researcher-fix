# Vibe Vibe Genesis Contribution System

This document defines the early participation system for project launches.

Goal:

- let the community participate before project token launch
- reward useful work and attention that creates real value
- avoid low-signal farming

## Core Thesis

Early participation should be earned through contribution, not just early speculation.

The system should reward:

- product help
- growth help
- customer help
- reusable asset creation
- trust and security help

The system should not reward empty noise.

## Core Objects

### 1. `Project Genesis Credits`

Non-transferable project-specific credits earned before token launch.

They represent:

- verified contribution
- proof of aligned support
- relative standing inside the project's early community

### 2. `Project Claim Rights`

A future right to claim project tokens if the project graduates and launches cleanly.

Claim rights are created from Genesis Credits only after:

- token readiness review passes
- launch parameters are locked
- utility is live

### 3. `Contributor Reputation`

Cross-project reputation attached to a contributor profile.

Use it to:

- weight future quest eligibility
- reduce sybil exposure
- rank high-trust contributors

## High-Signal Contribution Classes

- referred users who activate and retain
- bugs found and fixed
- onboarding improvements that increase activation
- customer interviews with usable extracted doctrine
- reusable prompts, workflows, or templates adopted by the team
- content that drives verified traffic or signups
- support moderation, docs, and education that measurably reduce founder load
- trust or security reviews that close real gaps

## Low-Signal Actions To Avoid Rewarding

- generic likes
- empty comments
- low-quality repost spam
- duplicate bug reports
- unverified referrals
- template submissions with no reuse
- AI-generated noise with no measurable result

## Reward Model

Each contribution should be scored on:

- evidence quality
- downstream impact
- founder acceptance
- reuse value
- durability of result

Suggested weighting:

- base reward for accepted task
- impact multiplier for measurable result
- durability multiplier after a time delay
- reputation multiplier for historically useful contributors

## Anti-Farm Controls

- non-transferable Genesis Credits
- contributor caps per epoch
- delayed reward release
- founder or reviewer signoff for high-value tasks
- sybil and duplicate detection
- slash or quarantine low-trust submissions
- minimum evidence requirement per quest

## Quest Types

### Growth Quests

- get qualified users
- source calls
- improve conversion copy
- produce proof content

### Product Quests

- identify bugs
- improve onboarding
- test workflows
- ship reusable assets

### Research Quests

- interview target users
- identify objections
- map competitor patterns
- extract doctrine packets

### Trust Quests

- review permissions
- find abuse cases
- improve safety copy
- close operational gaps

## Conversion Path

1. contributor completes verified quests
2. contributor earns Genesis Credits
3. project passes token readiness review
4. Genesis Credits convert into capped Claim Rights
5. Claim Rights convert into project token claims under the launch policy

## Important Boundaries

- Genesis Credits should not be tradable
- Claim Rights should be capped and policy-bound
- low-value projects should not mint fake reward expectations
- not all Genesis Credits must convert 1:1 into liquid tokens

## Clean Incentive Design

The most important behavior to preserve is:

- create value first
- receive standing second
- receive token exposure only after utility is real

That keeps the incubator aligned with builders rather than farmers.
